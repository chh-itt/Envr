# envr CLI v1.0 指标采集方案

本文档给出 `docs/cli/v1.0-definition.md` 中停止线指标的工程化落地方式，目标是：

- 复用现有 `envr_cli_metrics` 事件，不增加过多运行时开销；
- 产出稳定、可审计的 CI artifact；
- 让“是否进入维护期”可用数据自动判断。

---

## 1. 采集范围与原则

### 1.1 采集范围

- CLI 进程级事件：`phase=parse|dispatch|finish`（已存在）
- 命令级结果：成功/失败、退出码、错误码
- 关键体验指标：耗时、恢复率、契约稳定度、覆盖豁免数量

### 1.2 原则

- 不在 `stdout` 注入非契约日志（继续保持 JSON/porcelain 纯净）
- 指标源优先采用结构化 tracing + 测试/脚本结果
- 以“版本窗口”聚合，不以单次运行判断稳定性

---

## 2. 指标字典（v1.0 DoD 对应）

字段命名建议用于 `artifacts/cli-v1-health.json`。

### 2.1 任务成功率

- `bootstrap_success_rate`
  - 定义：首次执行引导路径（`install/use/init/check`）且最终 `finish.success=true` 的会话占比
  - 目标：`>= 0.95`
- `daily_run_success_rate`
  - 定义：`run|exec` 命令中 `finish.success=true` 的占比
  - 目标：`>= 0.99`

### 2.2 恢复能力

- `doctor_fix_recovery_rate`
  - 定义：`doctor --fix` 后在同一会话窗口内 `check|status|doctor` 变为成功的占比
  - 目标：`>= 0.85`

### 2.3 时间体验

- `time_to_first_success_p95_ms`
  - 定义：从会话内首个 CLI 调用到首个 `finish.success=true` 的耗时 P95
  - 目标：`<= 600000`（10 分钟）
- `offline_safe_latency_p95_ms`
  - 定义：`status/current/which/resolve` 的 `dispatch.elapsed_ms` 聚合 P95
  - 目标：`< 300`

### 2.4 契约稳定

- `breaking_contract_changes`
  - 定义：当前变更集中被 gate 判定为 breaking 的数量
  - 目标：`== 0`
- `governance_gate_passed`
  - 定义：`scripts/check_cli_governance_all.py` 是否通过
  - 目标：`true`
- `long_lived_exemptions`
  - 定义：超过 `due` 且仍未清理的 exemption 数量
  - 目标：`== 0`

### 2.5 复杂度控制

- `top_level_command_count`
  - 定义：`command_spec` 中 top-level 命令计数（别名不计）
  - 目标：冻结（与 v1.0 基线一致）
- `extension_over_new_command_ratio`
  - 定义：迭代中“扩展既有命令”需求占比
  - 目标：`>= 0.70`

---

## 3. 数据来源映射

| 指标 | 主数据源 | 备选/补充源 |
|------|----------|-------------|
| success_rate / latency | tracing `envr_cli_metrics` | 集成测试日志 |
| contract breaking | `check_cli_contract_gate.py` 结果 | schema diff 报告 |
| governance through | `check_cli_governance_all.py` | CI job status |
| exemptions | `schemas/cli/governance-index.json` | 自定义审计脚本 |
| command count | `crates/envr-cli/src/cli/command_spec.rs` | help registry 校验 |

注：若上线环境暂未统一采样会话 ID，可先在 CI 与集成测试环境内按 job/run 聚合，后续再扩展线上窗口统计。

## 5. CI 集成（已落地）

### 5.1 已接入步骤

在 `CLI Contract Governance` job 中已接入：

1. 读取上游产物与治理数据：
   - `artifacts/cli-contract-report.json`
   - `artifacts/cli-capabilities-report.json`
   - `schemas/cli/governance-index.json`
   - `crates/envr-cli/src/cli/command_spec.rs`
2. 生成报告：
   - `python scripts/generate_cli_v1_health_report.py --output artifacts/cli-v1-health.json`
3. 执行 hard guard 校验：
   - `python scripts/check_cli_v1_health.py --report artifacts/cli-v1-health.json`
4. 上传产物：
   - artifact `cli-v1-health`

### 5.2 脚本与契约文件

- 观测聚合脚本：`scripts/generate_cli_observed_metrics.py`
- 观测质量校验脚本：`scripts/check_cli_observed_metrics.py`
- 生成脚本：`scripts/generate_cli_v1_health_report.py`
- 校验脚本：`scripts/check_cli_v1_health.py`
- 报告 schema：`schemas/cli/cli-v1-health.schema.json`

### 5.3 可计算策略（已接入）

为避免关键指标长期 `pending`，生成脚本默认采用 **proxy 口径** 计算：

- `bootstrap_success_rate` / `daily_run_success_rate` / `doctor_fix_recovery_rate`：由 capability 覆盖映射计算；
- `time_to_first_success_p95_ms` / `offline_safe_latency_p95_ms`：由覆盖与 offline-safe 能力推导保守代理值；
- `extension_over_new_command_ratio`：由命令冻结基线与变更信号推导。

报告 `summary.metric_mode`：

- `proxy`：使用代理口径（默认 CI）；
- `observed`：通过观测文件或脚本参数传入后生成。

可通过以下参数覆盖为真实观测值：

- `--bootstrap-success-rate`
- `--daily-run-success-rate`
- `--doctor-fix-recovery-rate`
- `--time-to-first-success-p95-ms`
- `--offline-safe-latency-p95-ms`
- `--extension-over-new-command-ratio`

也可通过观测文件一次性覆盖：

- `python scripts/generate_cli_observed_metrics.py --metrics-jsonl artifacts/envr-cli-metrics.jsonl --output artifacts/cli-observed-metrics.json`
- `python scripts/generate_cli_v1_health_report.py --observed-metrics-json artifacts/cli-observed-metrics.json --output artifacts/cli-v1-health.json`

CI 已支持“优先 observed”：优先消费由 CI 真实运行生成的 `artifacts/envr-cli-metrics.jsonl`（tracing JSONL），若缺失则生成可审计的 CI smoke 样本，再走 observed 聚合。
当 observed 文件存在时，CI 会先执行质量门：

- 字段齐全（6 个核心指标 + `sample_size`）
- 值域合法（比例指标 `[0,1]`，耗时指标非负整数）
- 样本量下限（默认 `min_sample_size=20`）

并为可审计与收敛 gate 增加来源标识：

- `artifacts/cli-observed-metrics.json.observed_source`：
  - `ci_real_run`：来自 CI 真实运行采集的 tracing JSONL
  - `smoke_fixture`：来自 CI 生成的 smoke 样本
  - `flat_jsonl`：其它扁平 JSONL 输入

`main` 分支会强制要求 `observed_source == ci_real_run`（防止 smoke 抬高指标）。

并支持主分支收敛开关（当前默认开启）：

- `REQUIRE_OBSERVED_ON_MAIN=true` 时，`main` 分支若未产出 `artifacts/cli-observed-metrics.json` 则直接失败；
- 当前默认值为 `true`，用于确保 `main` 始终走 observed 口径。

### 5.4 增强集成（后做）

- 增加会话级埋点关联（`session_id`）后，计算更精确的 `time_to_first_success_p95_ms` 与恢复率

## 6. 阈值触发动作（建议）

### 6.1 阻断发布（hard fail）

任一满足即阻断：

- `breaking_contract_changes > 0`
- `governance_gate_passed == false`
- `long_lived_exemptions > 0`

### 6.2 预警不阻断（soft fail）

- `bootstrap_success_rate` 或 `daily_run_success_rate` 低于阈值
- `doctor_fix_recovery_rate` 连续两周下降
- `offline_safe_latency_p95_ms` 连续两周上升且超过阈值

## 8. 维护约定

- 任何指标字段改名/删除需给迁移说明；
- 阈值调整必须附回溯数据与评审记录；
- 指标用于决策，不替代人工评审（尤其是产品价值判断）。
