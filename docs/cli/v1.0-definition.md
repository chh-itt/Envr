# envr CLI v1.0 终态定义（草案）

本文档定义 `envr` CLI 的 **v1.0 终态边界**、**收敛路径** 与 **停止线指标**，用于避免功能无上限扩张，并给出进入“维护期”的客观判断标准。

相关规范：

- [输出契约（JSON / text）](./output-contract.md)
- [自动化输出矩阵（维护者清单）](./automation-matrix.md)
- [命令谱系（分层）](./commands.md)
- `schemas/cli/governance-index.json`（治理索引）

---

## 1. v1.0 的产品定位

`envr` CLI 的 v1.0 不是“命令尽可能多”，而是：

- 面向开发者与自动化系统的 **运行时环境控制平面（control plane）**；
- 在人机两侧都稳定：人类可理解、机器可依赖；
- 默认工作流闭环完整：`发现 -> 解释 -> 修复 -> 验证`。

### 1.1 终态目标（一句话）

在不破坏既有契约的前提下，让新用户在 3-5 分钟内完成核心路径，让自动化调用在升级后仍可预测运行。

---

## 2. 命令面终态边界（Command Surface）

v1.0 按任务价值分为三层，默认文档与帮助优先展示前两层。

### 2.1 Core 路径（必须“低认知负担”）

- 引导与安装：`install` / `use` / `init`
- 状态与解析：`status` / `current` / `which` / `resolve`
- 运行与校验：`run` / `exec` / `check`
- 诊断与恢复：`doctor` / `diagnostics export`

### 2.2 Automation 路径（必须“契约稳定”）

- `--format json`（单行 envelope + 稳定 `code`）
- `--porcelain`（已承诺的行协议）
- 离线与缓存：`cache index sync|status` / `bundle create|apply`

### 2.3 Expert 路径（可演进，但不破坏主路径）

- `project`, `profile`, `alias`, `template`, `hook`, `debug`, `rust` 等
- 原则：增强深度，不增加主路径复杂度

---

## 3. 版本收敛策略（避免无止境优化）

### 3.1 “功能冻结”定义

当满足第 6 节的停止线指标后，`v1.x` 进入“能力冻结 + 质量迭代”：

- 不新增 top-level 主命令；
- 优先通过现有命令参数、子命令、别名完成增强；
- 破坏性契约变更仅允许在 `v2` 讨论，不在 `v1.x` 落地。

### 3.2 新命令准入门槛（v1 阶段）

新增命令必须同时满足：

1. 对应一个独立且高频的用户任务（非“工程便利”）；
2. 无法通过现有命令组合清晰表达；
3. 具备完整 contract 落地（schema / matrix / tests / governance-index）；
4. 明确 owner 与维护成本评估（含 i18n / docs / test 预算）。

任一条件不满足，默认拒绝新增命令。

---

## 4. 体验优化主线（v1.x）

v1.x 的优化优先级不是“新增能力”，而是提升三件事：

1. **成功率**：首次上手可完成核心任务；
2. **可恢复性**：失败后能给出可执行下一步并快速恢复；
3. **可预测性**：脚本调用在版本升级中保持稳定。

### 4.1 具体原则

- 错误优先给可执行 `next_steps`（而非只给解释文本）；
- `doctor` 作为统一修复入口，逐步吸收高频失败场景；
- `status/check/doctor` 形成闭环，不让用户停在“知道错了但不会修”。

---

## 5. 发布节奏建议（v1.0 -> v1.x）

### 5.1 v1.0 前（收敛期）

- 集中清理帮助入口与命令分层呈现；
- 补齐核心任务链路的端到端回归；
- 明确并冻结 P0 合同面（JSON/porcelain）。

### 5.2 v1.0 后（维护期）

- 以质量与观测驱动迭代（成功率、恢复率、回归率）；
- 大功能需求先走 RFC，默认进入 v2 backlog；
- 定期做“复杂度预算审查”（命令数、帮助层级、文档长度）。

---

## 6. 停止线指标（Definition of Done）

满足以下指标连续 2-3 个版本（建议至少 8 周）即可判定“进入维护期”。

### 6.1 用户任务指标（必达）

- Bootstrap 首次成功率 >= 95%
- Daily run 成功率 >= 99%
- `doctor --fix` 后复测通过率 >= 85%
- `time_to_first_success` P95 <= 10 分钟

### 6.2 契约稳定指标（必达）

- P0 命令的破坏性 JSON 变更 = 0
- `check_cli_governance_all.py` 持续通过
- 自动化矩阵宣称项与测试覆盖一致（无长期豁免）

### 6.3 性能与可用性指标（建议）

- 离线安全命令（`status/current/which/resolve`）P95 < 300ms
- 关键命令失败路径具备稳定 `code` + 可执行 hint
- `--format json` stdout 单行率 100%（无日志污染）

### 6.4 复杂度控制指标（必达）

- top-level 主命令数量冻结（别名不计入）
- 新需求中 >= 70% 通过现有命令扩展实现
- `help` 首页核心任务入口稳定且不膨胀

---

## 7. 指标与治理映射（建议落地）

可把指标映射到现有治理资产：

- 合同稳定：`docs/cli/output-contract.md` + `schemas/cli/*` + gate scripts
- 覆盖一致：`docs/cli/automation-matrix.md` + `governance-index.json`
- 命令边界：`docs/cli/commands.md` + `command_spec.rs` 的 capabilities

建议新增一个轻量报告（可复用现有 artifact 机制）：

- `artifacts/cli-v1-health.json`
- 字段包含：任务成功率、恢复率、契约变更数、豁免数、延迟分位数

---

## 8. 非目标（v1.x 明确不做）

- 为了“命令齐全”而新增 top-level 命令；
- 在 `v1.x` 引入破坏性 envelope 语义变化；
- 将人类可读文案当成自动化契约字段依赖；
- 以牺牲一致性为代价的短期功能捷径。

---

## 9. 评审与执行建议

建议每个迭代以本文件作为评审模板，回答四个问题：

1. 本次改动是否提升了核心任务成功率？
2. 是否增加了命令面复杂度？若增加，是否通过准入门槛？
3. 是否触及契约风险？若触及，是否完成迁移说明与 gate 校验？
4. 是否有可观测证据支持上线（测试、指标、回归报告）？

若以上四问无法给出明确正面结论，默认不进入发布分支。
