# 缺陷分级与处理机制（Bug triage）

本文档统一 **envr** 仓库内缺陷的优先级定义、响应与修复期望、以及关闭与回归验证要求。适用于 GitHub Issues、内部清单及发布前的已知问题收敛。

---

## 1. 目标

- 任意新缺陷均可归入 **blocker / critical / major / minor** 之一。
- 处理节奏可预期（SLA 为**团队承诺基线**，可根据里程碑临时调整并在 Issue 中注明）。
- 关闭前具备**可验证**的修复或明确的不修复结论，并在可能时补充**回归**（自动化测试或矩阵手测项）。

---

## 2. 严重级别定义

### 2.1 Blocker（阻断）

**定义**：导致产品在**目标环境**下**完全不可用**，且无合理变通手段。

**典型情形**（示例）：

- 安装或首次启动后，CLI 在支持平台上**必然崩溃**或无法执行任何子命令。
- 发布包**无法安装/解压/校验**，或官方文档声明的核心路径**全体失败**。
- **数据损坏**或**未授权的安全后果**（若与安全相关，同时按安全披露流程处理）。

**非 Blocker**：单一可选功能失败、仅影响 GUI 而 CLI 可用、仅影响非默认镜像/网络环境。

### 2.2 Critical（严重）

**定义**：**核心路径**存在严重故障，但存在**有限变通**（例如可用 CLI 代替 GUI），或影响**大量用户**的高频场景。

**典型情形**：

- `install` / `use` / `list` 等主命令在常见配置下**稳定失败**。
- Shim 在文档约定的使用方式下**错误路由**到错误运行时或静默失败。
- GUI 在主流 Windows 配置下**无法启动**，且产品主推图形安装流程时。

### 2.3 Major（重要）

**定义**：功能**明显错误**或体验**严重受损**，但不阻断整条产品线；或**回归**（曾经正常的行为被破坏）。

**典型情形**：

- 特定语言 runtime 或特定版本组合下失败，且有替代版本可用。
- JSON 输出与文档契约不一致，导致脚本集成方需要额外处理。
- 性能问题导致操作**明显不可接受**（如无故卡死数分钟且无进度反馈）。

### 2.4 Minor（一般）

**定义**：**边缘场景**、文案/排版、非阻塞性告警、对体验影响**较小**的问题。

**典型情形**：

- 帮助文案错误、日志级别不当、浅色主题下对比度略差。
- 极罕见平台或需特殊环境复现的问题。

### 2.5 与安全、隐私相关

- 一律**至少**按 **Critical** 起评，并完成影响面评估。
- 若存在利用条件与数据风险，应按项目安全策略**限制公开细节**，修复后再披露（如有单独安全策略文档，以其为准）。

## 3. 响应与处理 SLA（建议基线）

以下为**工作日**维度的团队承诺基线；节假日与集中发布周可在里程碑说明中临时调整。

| 级别 | 首次响应 | 修复或明确计划 | 说明 |
|------|----------|----------------|------|
| **Blocker** | 1 个工作日内 | 3 个工作日内给出**可安装/可运行**的修复或回滚方案 | 需指定负责人；每日同步状态 |
| **Critical** | 2 个工作日内 | 7 个工作日内修复或发布带规避说明的**补丁/小版本** | 若依赖上游，需在 Issue 记录外部跟踪链接 |
| **Major** | 5 个工作日内 | 当前里程碑内纳入排期；无法纳入则**降级为已知问题**并写入 `docs/release/KNOWN-ISSUES.md` | 需在 Issue 标注目标版本或 `wontfix` 理由 |
| **Minor** | 10 个工作日内 | 按迭代容量安排；可合并至“体验打磨”类里程碑 | 允许长期 backlog，但避免无限期无响应 |

**首次响应**：在 Issue 中留下分级结果、是否需要更多信息、是否已复现。  
**修复或明确计划**：合并 PR、发布版本、或书面说明不修复/推迟及原因。

---

## 4. 工作流程（闭环）

1. **登记**：新建 Issue，模板中应包含：环境（OS、envr 版本）、复现步骤、期望与实际行为、日志片段（若有）。
2. **分级**：维护者或值班人打上 `severity/blocker` 等标签（或在本仓库约定等效方式），并写入一句分级理由。
3. **处理**：关联 PR；Blocker/Critical 优先 code review 与合并。
4. **验证**：
   - 修复方或报告方在 Issue 中说明**验证方式**（命令、场景）。
   - **推荐**：为 Major 及以上补充**自动化测试**；若暂不可行，在 `docs/qa/regression-matrix.md`（T902）或发布手测中增加**可打勾**项。
5. **关闭**：使用 `closes #123` 或关闭时引用 PR；若属已知限制，关闭前更新 `KNOWN-ISSUES.md` 或发行说明。

**升级**：若 Critical 在 SLA 内无负责人或无法取得进展，维护者应将 Issue 置顶并同步到发布/项目管理渠道（与团队习惯一致即可）。

---

## 5. 与发布、测试文档的关系

- **已知问题**：面向用户的条目应同步到 `docs/release/KNOWN-ISSUES.md`（或对应版本发行说明）。
- **回归**：与 T902「回归测试矩阵」对齐；Blocker/Critical 的修复**原则上**应有自动化或矩阵中的明确手测项。
- **覆盖率**：质量门禁以 `cargo envr-cov`（见 `.cargo/config.toml`）为准；缺陷修复若落在门禁范围外 crate，应在 PR 中说明测试策略。

---

## 6. Issue 标签约定（建议）

可在 GitHub 中配置下列标签（名称可按仓库习惯微调，含义与本文件一致即可）：

| 标签 | 含义 |
|------|------|
| `severity/blocker` | 本节 2.1 |
| `severity/critical` | 本节 2.2 |
| `severity/major` | 本节 2.3 |
| `severity/minor` | 本节 2.4 |
| `needs-triage` | 尚未分级 |
| `needs-repro` | 待复现或信息不足 |

---

## 7. 修订

本文档随流程演进而更新；重大变更应在团队内同步，并在 `tasks.md` 或变更说明中留痕。
