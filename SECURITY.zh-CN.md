# 安全策略

English | [简体中文](SECURITY.zh-CN.md)

## 支持版本

`envr` 目前处于 pre-1.0 阶段，尚未维护长期支持分支。

在这个阶段，安全修复预计优先进入默认开发分支。
我们通常将以下版本视为安全报告与修复的支持范围：

| 版本 | 是否支持 |
|---|---|
| 当前默认分支 / 最新未发布工作 | 支持 |
| 最新 tag 版本（如果存在） | 尽力支持 |
| 更旧的 tag / 历史 commit | 不支持 |

如果漏洞报告针对较旧版本，维护者可能会先要求你确认其是否仍能在最新代码上复现。

## 报告漏洞

请不要在公开 GitHub issue 中报告安全漏洞。

请通过以下私下渠道报告：

1. 如果仓库启用了 GitHub Security Advisory / 私有漏洞报告，请优先使用。
2. 如果无法使用私有 advisory，请在公开披露前先通过私下渠道联系维护者。

如果你不确定问题是否安全敏感，建议先私下报告。

## 报告中应包含的内容

请尽量提供以下信息：

- 受影响的 `envr` 版本、commit 或分支
- 操作系统与架构
- 涉及的 runtime / provider
- 与问题相关的配置细节
- 精确的复现命令或步骤
- 在安全情况下可共享的 PoC、示例 archive 或 mirror / index
- 影响评估：代码执行、路径劫持、归档穿越、校验绕过、权限边界混淆、本地 secret 暴露、拒绝服务等
- 问题是否依赖恶意 mirror、被攻破的上游、本地文件系统访问或不受信任的项目文件

## `envr` 中的安全敏感区域

由于 `envr` 是 runtime 管理器，我们特别关注以下区域：

- 远程二进制与元数据下载
- mirror 与上游 index 选择
- 校验和与完整性验证
- 归档解压与目标路径处理
- shim 生成与可执行文件解析
- `PATH` 操作与 shell 集成
- 项目本地配置（例如 `.envr.toml`）
- 本地缓存复用与缓存恢复行为
- 诊断导出与环境 / 路径信息泄露

涉及以上任一区域的报告都应私下提交。

## 披露策略

请给维护者合理时间来调查并准备修复后再公开披露。

我们的首选流程是：

1. 收到私下报告
2. 维护者确认影响并完成分诊
3. 准备并审核修复
4. 发布修复或 advisory
5. 协调公开披露

我们希望报告者在修复或缓解方案可用之前，不要提前公开利用细节，除非与维护者协商一致。

## 期望响应时间

当前响应目标为 best effort：

- 初次确认：**7 个自然日**内
- 首轮分诊更新：**14 个自然日**内
- 对已确认问题的状态更新：在合理间隔内持续提供

涉及上游 runtime provider、mirror、归档格式或跨平台行为的复杂问题，可能需要更长时间。

## 范围说明

### 属于范围内

以下问题通常会被视为安全漏洞：

- 归档解压路径穿越或覆盖到预期目录之外
- 校验和验证绕过或完整性降级
- 对未验证 mirror 或元数据的不安全信任，导致非预期二进制执行
- 可能导致意外命令劫持的 shim 或 `PATH` 行为，超出文档中描述的信任假设
- 可能触发特权或意外执行流的项目本地配置处理
- 通过诊断或导出的配置造成的信息泄露，超出预期

### 通常不在范围内或严重性较低

以下情况仍可能是 bug，但不一定单独作为安全问题处理：

- 需要用户主动运行一个明显恶意的本地二进制
- 完全由恶意本地管理员或已被攻陷的机器解释的行为
- 仅限于删除用户自己的缓存或 runtime root 的拒绝服务
- 仅因不支持的平台或不支持的自定义 mirror 导致的故障，且不涉及安全边界绕过

## 用户侧加固建议

在 `envr` 进入更稳定的发布流程之前，建议用户采取常规防护：

- 优先使用可信网络与可信 mirror
- 在不受信任仓库中使用项目本地 `.envr.toml` 前先检查其内容
- 不要让不同信任域共享可写的 runtime root
- 将 runtime root 与 cache 目录设置为仅允许预期用户写入
- 当官方发布包可用时，请验证 release artifact 与 checksum

## 依赖漏洞策略

CI 与 release 打包应运行 `cargo deny check`，包括 RustSec advisory。
`deny.toml` 中任何临时忽略项都必须在 [`docs/deps/security-and-licenses.md`](docs/deps/security-and-licenses.md) 中记录风险接受原因与复审计划。

当前可接受的 advisory 忽略仅限于 GUI 依赖栈中没有安全升级路径的传递性问题。它们必须在 release 前以及 GUI 依赖栈升级时复审。

## 安全更新

与安全相关的修复会在适用时写入 release notes。
如果修复需要运维上的后续动作，维护者也应记录推荐的缓解或清理步骤。
