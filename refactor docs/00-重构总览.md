# envr 重构总览（v3）

## 1. 目标与边界

### 1.1 重构目标
- 将项目统一命名为 `envr`，替代 `wanxiang_toolkit / wanxiang` 命名。
- 继承“旧项目”的完整功能覆盖能力。
- 继承“未完成的重构项目”的 workspace 分层架构。
- 重点解决：高耦合、扩展慢、测试困难、多人协作难的问题。

### 1.2 明确边界
- 不做兼容迁移：不兼容旧配置目录、旧 CLI 名、旧路径结构（按你要求）。
- 不以“尽快上线”为第一目标，而是以“可持续迭代”为第一目标。
- v3 优先 Windows 稳定，macOS/Linux 作为后续里程碑。

## 2. 现状评估（两版取舍）

### 2.1 旧项目（用于借鉴）
- 优点：
  - 功能全，语言覆盖广（Node/Python/Java/Go/Rust/PHP/Deno/Bun）。
  - GUI/CLI/下载/镜像/shim 都有实践经验。
- 痛点：
  - 单包内模块过多，边界不清。
  - GUI、CLI、核心逻辑相互耦合，复用困难。
  - 功能迭代时牵一发而动全身。

### 2.2 未完成的重构项目
- 优点：
  - workspace + 多 crate 拆分方向正确（`wx-core`、`wx-cli`、`wx-shim`、`wx-ui`）。
  - 具备更好的可测试性和平台抽象基础。
- 痛点：
  - 业务尚未补齐，命令和语言支持完成度不一致。
  - `wx-core` 对外 re-export 面过大，接口层仍偏重。
  - 文档与实现节奏不同步，容易“设计超前但落地慢”。

## 3. v3 总体策略

- 架构策略：沿用 workspace 思路，但进一步“按能力分层 + 按语言插件化”。
- 交付策略：以 CLI 为先（功能收敛快、可测试强），GUI 第二阶段对齐。
- 质量策略：先建“最小可验证链路”（安装-切换-执行-卸载-诊断），再扩语种。
- 风险策略：把 shim、下载、镜像当作基础设施优先稳定。

## 5. 首阶段落地建议（你可直接用）

- M1（核心基建）：workspace、统一错误模型、配置系统、下载引擎骨架。
- M2（CLI 主链路）：Node/Python/Java 的 install/use/list/current/uninstall/which/remote/doctor。
- M3（shim 稳定）：全局 + 项目级 `.envr.toml` 解析，PATH 接入，跨平台脚本注入。
- M4（GUI 接入）：环境中心、下载面板、基础设置页，复用 CLI 同一核心服务。
- M5（扩语种）：Go/Rust/PHP/Deno/Bun 与高级功能（alias/profile/import/export）。
