# envr 实施路线图（建议）

## Phase 0：工程基线（1 周）

- 建立 workspace 骨架与 crate 命名（envr-*）。
- 统一错误模型、日志、配置目录约定。
- CI 基线：`fmt + clippy + test`，接入覆盖率统计（总体覆盖率目标 ≥ 80%）。

## Phase 1：核心链路（2-3 周）

- 实现 Node/Python/Java 的最小全链路：
  - `install/use/list/current/uninstall/which/remote/doctor`
- 打通下载引擎 + 镜像策略 + shim 全局切换。
- 完成 CLI JSON 输出契约。

## Phase 2：项目级能力（1-2 周）

- `.envr` / `.envr.local` 解析与优先级覆盖。
- shell 集成与自动切换（先 Windows，后 Unix）。
- 黑盒测试覆盖项目目录上溯解析逻辑。

## Phase 3：GUI 接入（2 周）

- 环境中心 + 下载面板 + 设置页。
- 复用 core API，不重复实现业务。
- 处理 GUI 异步任务状态和错误映射。

## Phase 4：扩展语言与高级命令（持续）

- Go/Rust/PHP/Deno/Bun 逐步接入。
- 补齐 alias/profile/import/export 等高级命令。
- 增加性能与稳定性测试（大批量版本列表、并发下载压测）。
- 测试类型要求：单元测试 + 集成测试 + 属性测试（proptest）+ 平台测试（Windows/macOS/Linux），整体覆盖率 ≥ 80%。

## 每阶段验收门槛（DoD）

- 功能：主链路跑通且有集成测试。
- 架构：无跨层依赖违规（可用 deny 清单）。
- 质量：关键路径错误码明确，日志可追踪。
- 文档：命令文档与设计文档同步更新。
