# envr 依赖选择与原则

本文件用于约束各 crate 的依赖选择，避免出现依赖膨胀、特性冲突或平台兼容性问题。

## 1. 总体原则

- **最小足够原则**：优先选择功能刚好满足需求的轻量库，而不是“全家桶”。
- **平台优先级**：优先保证 Windows 体验，其次 macOS，再到 Linux（但不破坏跨平台能力）。
- **长期维护**：优先选 API 稳定、有活跃维护、在 Rust 生态中被广泛使用的库。
- **安全与性能**：避免引入不必要的 `unsafe`、大型运行时或重度宏黑魔法。

## 2. 各 crate 依赖策略（概要）

- `envr-error`：
  - 错误模型：优先使用 `thiserror`。
  - 不允许依赖 GUI / CLI / 下载相关库。

- `envr-config`：
  - 配置格式：使用 `toml` 解析 `.envr.toml` / `.envr.local.toml`。
  - 建议：`toml` + `serde` 组合，禁止自研解析器。

- `envr-download`：
  - HTTP 客户端：优先基于 `reqwest`（rustls 后端，关闭不必要默认特性）。
  - 压缩与解压：`zip` / `flate2` / `tar` 等按需开启。
  - 并发：基于 `tokio` + `futures`，谨慎引入额外 runtime。

- `envr-mirror`：
  - 仅依赖 `serde` / `serde_json` / `toml` 等通用库。
  - 不直接访问网络，由 `envr-download` 提供 HTTP 能力。

- `envr-shim-core`：
  - 尽量零第三方依赖，仅必要时引入 `toml`（解析 `.envr.toml`）和少量工具库。

- `envr-runtime-*`：
  - 按语言需求引入解析或索引相关依赖（如 `regex`），禁止直接依赖 GUI/CLI。

- `envr-cli`：
  - CLI 解析：使用 `clap`（精简特性集）。
  - i18n：复用核心的 i18n 层，不额外引入多套方案。

- `envr-gui` / `envr-ui`：
  - GUI 框架：基于 `iced 0.14`，按平台选择合适 backend（优先 `gl`）。
  - 避免引入与 `iced` 冲突的事件循环或渲染依赖。

## 3. 依赖管理规范

- workspace 统一依赖：
  - 在 `workspace.dependencies` 中集中声明常用库版本（如 `serde`、`tokio`、`reqwest`、`thiserror`、`toml`、`clap` 等），避免版本漂移。
- 特性管理：
  - 显式启用需要的 feature（如 `reqwest` 的 `rustls-tls`、`stream`、`gzip`）。
  - 禁用 `native-tls` 等增加额外系统依赖的特性，除非有强需求。
- 平台差异：
  - 使用 `cfg(target_os = "...")` 控制平台相关依赖与实现。
  - Windows 专属依赖（如 `windows-sys`）放在专门的 crate 或模块中。

## 4. 测试与工具依赖

- 测试：
  - 单元测试与集成测试可引入 `proptest`、`tempfile`、`assert_cmd` 等。
  - 测试依赖统一在 `workspace.dependencies` 或 `dev-dependencies` 中声明。
- 开发工具：
  - 允许按需使用 `criterion` 做性能基准，但不强制。

## 5. 引入新依赖的流程（建议）

- 明确用途与替代方案比较。
- 评估维护状况、最近发布频率、Issue/PR 活跃度。
- 评估二进制体积、内存占用与启动时间影响。
- 在提交前更新本文件中对应 crate 的“依赖策略”小节。
